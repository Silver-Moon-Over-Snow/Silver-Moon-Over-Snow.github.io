### KISS 原则

KISS = **Keep it simple, stupid**。强调设计和实现应尽量保持简单，避免不必要的复杂性。

### 脚本模板：建立 100 个目录

文件名可写为 `make_dirs.sh`：

```bash
#!/bin/bash

for i in $(seq 1 100)
do
    mkdir -p DIR$i
done
```

运行方式：

```bash
chmod +x make_dirs.sh
./make_dirs.sh
```

> 也可以不添加执行权限：`bash make_dirs.sh`

## 四、MPI

### 基本概念

- MPI = **Message Passing Interface**
- 常见编程模型：SPMD = **Single Program Multiple Data**
- MPI 程序必须初始化和结束：

```cpp
MPI_Init(&argc, &argv);
MPI_Finalize();
```

- **进程**拥有独立地址空间和系统资源
- **线程**共享同一进程的地址空间和资源

### 通信类型

| 类别 | 函数 | 说明 |
|------|------|------|
| 点对点 | `MPI_Send`、`MPI_Recv` | 阻塞通信 |
| 点对点 | `MPI_Isend`、`MPI_Irecv` | 非阻塞通信 |
| 集合通信 | `MPI_Bcast` | root 向所有进程广播 |
| 集合通信 | `MPI_Scatter` | root 把数组分块发给各进程 |
| 集合通信 | `MPI_Gather` | 各进程数据收集到 root |
| 集合通信 | `MPI_Reduce` | 把所有进程数据规约到 root |
| 集合通信 | `MPI_Allreduce` | 规约后把结果分发给所有进程 |

### 同步与通信器

- `MPI_Barrier(comm)`：等待同一通信域中所有进程到达
- `MPI_Comm_split(comm, color, key, &newcomm)`：
  - `color` 决定分到哪个新通信器
  - `key` 决定新通信器中的 rank 排序
- `MPI_Comm_create()`：用 group 创建新的 communicator

### 死锁

MPI 死锁常见原因：

- 进程互相等待
- 阻塞通信顺序不合理
- 循环等待（最关键）
- 发送和接收不匹配

> [!warning] 易错辨析
> 死锁必要条件中最典型的是 **D. 各个进程之间存在循环等待**。虽然 A. "持有资源并等待其他资源" 也属于一般死锁 Coffman 条件之一，但 MPI 通信死锁的核心特征是循环等待。

### 编程模板：两个进程非阻塞互发

进程 0 发 `int` 给进程 1，进程 1 发 `double` 给进程 0（2023 年真题题型）：

```cpp
// mpi_isend_irecv.cpp
#include <mpi.h>
#include <iostream>

int main(int argc, char** argv) {
    MPI_Init(&argc, &argv);

    int rank, size;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    if (size != 2) {
        if (rank == 0)
            std::cout << "Please run with 2 processes." << std::endl;
        MPI_Finalize();
        return 0;
    }

    MPI_Request reqs[2];
    MPI_Status stats[2];

    if (rank == 0) {
        int send_data = 1;
        double recv_data = 0.0;
        MPI_Isend(&send_data, 1, MPI_INT, 1, 0, MPI_COMM_WORLD, &reqs[0]);
        MPI_Irecv(&recv_data, 1, MPI_DOUBLE, 1, 1, MPI_COMM_WORLD, &reqs[1]);
        MPI_Waitall(2, reqs, stats);
        std::cout << "Rank 0 received " << recv_data << std::endl;
    } else if (rank == 1) {
        double send_data = 0.5;
        int recv_data = 0;
        MPI_Isend(&send_data, 1, MPI_DOUBLE, 0, 1, MPI_COMM_WORLD, &reqs[0]);
        MPI_Irecv(&recv_data, 1, MPI_INT, 0, 0, MPI_COMM_WORLD, &reqs[1]);
        MPI_Waitall(2, reqs, stats);
        std::cout << "Rank 1 received " << recv_data << std::endl;
    }

    MPI_Finalize();
    return 0;
}
```

编译运行：

```bash
mpicxx mpi_isend_irecv.cpp -o mpi_isend_irecv
mpirun -np 2 ./mpi_isend_irecv
```

## 五、OpenMP

### 高频考点

| 考点 | 指令/函数 | 说明 |
|------|-----------|------|
| 并行区 | `#pragma omp parallel` | 创建并行区 |
| 循环分配 | `#pragma omp for` | 将循环迭代分配给线程 |
| 临界区 | `#pragma omp critical` | 同一时间只一个线程执行（多行代码） |
| 原子操作 | `#pragma omp atomic` | 单条语句的原子更新 |
| 屏障 | `#pragma omp barrier` | 等待所有线程 |
| 任务等待 | `#pragma omp taskwait` | 等待当前任务创建的子任务完成 |
| 单线程执行 | `#pragma omp single` | 只让一个线程执行 |
| 主线程执行 | `#pragma omp master` | 只让 master 线程执行 |
| 私有变量 | `private` | 每个线程拥有独立副本 |
| 共享变量 | `shared` | 所有线程共享 |
| 规约 | `reduction(+:sum)` | 自动汇总各线程局部值 |
| 设置线程数 | `export OMP_NUM_THREADS=4` | 环境变量方式 |
| 设置线程数 | `omp_set_num_threads(4)` | 运行时函数方式 |
| 获取线程 ID | `omp_get_thread_num()` | 返回当前线程编号 |
| 获取线程数 | `omp_get_num_threads()` | 返回并行区实际线程数 |

### OpenMP 三部分

1. **编译器指令**（`#pragma omp ...`）
2. **运行时库函数**（`omp_*` 系列）
3. **环境变量**（`OMP_NUM_THREADS` 等）

> [!warning] 易错辨析（OpenMP 功能边界）
> OpenMP 的核心能力包括：共享内存并行、循环工作分配、同步互斥规约、动态线程控制。**"和 MPI 一起使用"** 属于 OpenMP + MPI 混合并行编程的应用方式，不是 OpenMP 指令本身直接实现的功能。

### 调度

| 调度方式 | 行为 |
|----------|------|
| `schedule(static, chunk_size)` | 按固定大小连续迭代块分配，`chunk_size` 指每个连续迭代块的大小 |
| `dynamic` | 线程做完一个块再取下一个，适合负载不均 |
| `guided` | 块大小逐渐减小 |

### 伪共享

伪共享是多个线程修改**不同变量**，但这些变量落在**同一个缓存行**中，导致缓存行频繁失效，性能下降。

> [!warning] 易错辨析
> 伪共享的核心是"不同变量 + 同一缓存行 + 写入冲突"，不是读同一个变量导致的性能问题。

### 编程模板：4 线程静态调度求和

1 到 10 求和，4 个线程，`schedule(static, 2)`，并行区内输出每个线程处理的数和最终结果（2024 年真题）：

```cpp
// openmp_sum.cpp
#include <iostream>
#include <omp.h>
using namespace std;

int main() {
    const int n = 10;
    int a[n];
    for (int i = 0; i < n; ++i) a[i] = i + 1;  // {1, 2, ..., 10}

    int sum = 0;
    omp_set_num_threads(4);

    #pragma omp parallel
    {
        int tid = omp_get_thread_num();
        int local_sum = 0;

        #pragma omp for schedule(static, 2)
        for (int i = 0; i < n; ++i) {
            local_sum += a[i];
            #pragma omp critical
            {
                cout << "Thread " << tid
                     << " handles a[" << i << "] = " << a[i] << endl;
            }
        }

        #pragma omp critical
        { sum += local_sum; }

        #pragma omp barrier

        #pragma omp single
        { cout << "Final sum = " << sum << endl; }
    }

    return 0;
}
```

编译运行：

```bash
g++ -fopenmp openmp_sum.cpp -o openmp_sum
./openmp_sum
```

**静态调度分配说明：**

`schedule(static, 2)` 表示每次连续分配 2 个循环迭代给一个线程：

| 线程 | 处理的数 |
|------|----------|
| Thread 0 | 1, 2, 9, 10 |
| Thread 1 | 3, 4 |
| Thread 2 | 5, 6 |
| Thread 3 | 7, 8 |

预期最终结果：`Final sum = 55`

## 六、GPU / CUDA

### 基本概念

| 概念 | 说明 |
|------|------|
| GPU | **Graphics Processing Unit** |
| SIMD | **Single Instruction Multiple Data** |
| Warp | 通常包含 **32** 个线程 |
| 线程组织 | Grid → Block → Thread |
| Block 内 | 线程可以协作（共享内存、同步） |
| Block 间 | 通常不能直接同步 |
| Grid/Block | 可组织成一维、二维、三维 |

### 全局线程索引

一维 CUDA 全局索引：

```cpp
int idx = threadIdx.x + blockIdx.x * blockDim.x;
```

### kernel 调用参数

```cpp
kernel<<<gridDim, blockDim, sharedMemBytes, stream>>>();
```

| 参数 | 含义 |
|------|------|
| `x`（gridDim） | grid 配置，block 数 |
| `y`（blockDim） | block 配置，每个 block 的线程数 |
| `m`（sharedMemBytes） | 动态共享内存大小，单位字节 |
| `s`（stream） | CUDA stream |

### CUDA 降低延迟的方法

- 使用**共享内存**代替全局内存（片上存储，延迟更低）
- 使用**常量缓存**等手动管理缓存
- 提高活跃 warp 数**隐藏**延迟
- 使用 **stream** 让多个 kernel 或数据传输并发

> [!warning] 易错辨析（延迟优化）
> - A. 使用共享内存 → ✓ 直接降低访存延迟
> - B. 增加线程块线程数 → 这是"隐藏延迟"而非"减少延迟"
> - C. 减少并行度 → ✗ 反而可能降低性能
> - D. 手动管理缓存（常量缓存等） → ✓ 可降低访问延迟

### 相关 API

| API | 说明 |
|-----|------|
| `cudaStreamCreate()` | 创建 stream，使设备可并行组织多个任务 |
| `cudaMemcpyAsync()` | 异步拷贝 |
| `cudaDeviceSynchronize()` | 等待设备任务完成 |

### GPU 性能（A100）

- 单精度 FP32：约 **19.5 TFlops**
- 双精度 FP64：约 **9.7 TFlops**

### CUDA Hello World 模板

2 个 block，2 个线程（2024 年真题）：

```cpp
// hello_cuda.cu
#include <cstdio>
#include <cuda_runtime.h>

__global__ void helloKernel() {
    int global_id = blockIdx.x * blockDim.x + threadIdx.x;
    printf("Hello World from block %d, thread %d, global thread %d\n",
           blockIdx.x, threadIdx.x, global_id);
}

int main() {
    helloKernel<<<2, 2>>>();
    cudaDeviceSynchronize();
    return 0;
}
```

编译运行：

```bash
nvcc hello_cuda.cu -o hello_cuda
./hello_cuda
```

预期输出 4 行（顺序不固定），每个 block 的 thread 0 和 thread 1 各输出一行。

## 七、并行性能

### 强可扩展性

当处理器数量加倍，问题规模不变，执行时间接近减半，说明程序具有强可扩展性。

### 弱可扩展性

处理器数量增加时，每个处理器承担的问题规模保持不变，整体运行时间基本不变。

### Amdahl 定律

公式：

```text
Speedup = 1 / (S + P / N)
```

其中：

- `S`：串行部分
- `P`：可并行部分
- `N`：处理器数量

如果 10% 串行、90% 可并行，理论最大加速比：

```text
1 / 0.1 = 10
```

## 八、易错题专项辨析

### 第 7 题：C++ 浮点类型

C++ 的浮点类型包括 `float`、`double`、`long double`。**`long long` 是整数类型**，`size_t` 通常也是无符号整数类型别名。若单选题，常见命题意图选 **C. long long**；严格说 C 和 D 都不是浮点类型。

### 第 13 题：MPI 死锁条件

MPI 死锁常见原因：互相等待、阻塞顺序不匹配、循环等待。若单选，最典型答案是 **D. 各个进程之间存在循环等待**。从一般 Coffman 条件看，A 也属于必要条件之一。

### 第 18 题：OpenMP 功能边界

OpenMP 核心能力：共享内存并行、循环工作分配、同步互斥规约、动态线程控制。**"和 MPI 一起使用"是混合编程的应用方式，不是 OpenMP 指令本身的功能。**

### 第 27 题：CUDA 延迟优化

- ✓ 共享内存代替全局内存 → 降低访存延迟
- ✗ 增加线程块线程数 → 隐藏延迟，非减少延迟
- ✗ 减少并行度 → 通常不利于性能
- ✓ 手动管理缓存 → 可降低延迟
